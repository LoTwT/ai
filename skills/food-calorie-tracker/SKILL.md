# 食物热量追踪 Skill

## 概述

这个 skill 帮助用户通过拍照记录每日饮食，自动识别食物、估算热量，并生成美观的汇总长图。

## 工作流程

### 第一步：识别食物并估算热量

食物识别和热量估算**必须**先读取同级的 `../food-calorie-estimator/SKILL.md`，再遵循其中的方法论。本 skill 不包含独立的估算逻辑，所有估算由 estimator 的流程驱动。

如果 estimator 未加载，遵守以下三条底线原则：
1. 先有可量化份量，再算热量 — 固体先估重量，饮品先估容量；只有包装/品牌已给出精确总热量时可直接使用。无容器参照时的默认份量：一人份主食 200g，一份菜 250-300g，饮品按大杯算
2. 不确定时取高值 — 份量、烹饪方式、品种都取偏高的
3. 热量影响大的差异才问用户 — 肉类品种、烹饪方式（炒 vs 炸）、饮品糖度必须问；蔬菜品种、调味咸淡不问

估算完成后，判断餐次（优先使用用户明确说明的餐次；如果用户没说，根据当前对话时间或上下文推断），以简洁的列表形式告知用户：

```
识别结果：
- 白米饭 约200g | 230 kcal
- 红烧排骨 约150g | 375 kcal
- 炒青菜 约100g | 35 kcal
- 拿铁咖啡（大杯，全脂奶）约480ml | 220 kcal（已按较高值估算）
合计：860 kcal

已记录为【午餐】。如有不准确的地方请告诉我修正。
```

用户可能会纠正识别结果（如"那个不是排骨是红烧肉"、"饭量没那么多大概150g"）。食物种类纠正需按 estimator 流程重新查热量数据并重算；份量纠正则保持每100g热量不变，按新份量重算。

### 第二步：累积记录

在同一对话中，用户可能分多次发送不同餐次的食物图片。每次识别后都追加到当天的记录中，并告知当前累计情况：

```
今日累计：
早餐：520 kcal
午餐：640 kcal
合计：1160 kcal
```

如果用户发完图片后没有主动要求生成汇总，不需要自动生成——继续等待用户的下一步指令即可。用户可能还要补发其他餐次。

### 第三步：生成汇总长图

当用户要求生成汇总（如"生成今日汇总"、"出图"、"汇总一下"、"生成长图"），使用 `scripts/generate_summary.py` 脚本生成最终的长图。

#### 准备数据

将当天的饮食记录整理为 JSON 格式，保存到一个临时文件。每个 meal 下面有自己的 `images` 字段，这样照片和餐次的对应关系不会错乱：

```json
{
  "date": "2026-03-15",
  "meals": [
    {
      "meal_type": "早餐",
      "items": [
        {"name": "白粥", "weight_g": 300, "calories": 140},
        {"name": "油条", "weight_g": 80, "calories": 390}
      ],
      "images": ["/path/to/breakfast_photo.jpg"]
    },
    {
      "meal_type": "午餐",
      "items": [
        {"name": "白米饭", "weight_g": 200, "calories": 230},
        {"name": "红烧排骨", "weight_g": 150, "calories": 375}
      ],
      "images": ["/path/to/lunch_photo1.jpg", "/path/to/lunch_photo2.jpg"]
    }
  ]
}
```

关于日期：使用当天的实际日期，格式为 `YYYY-MM-DD`。如果用户明确指定了日期（如"这是昨天的"），按用户说的来。

关于图片路径：记录用户每次上传的食物图片的实际路径，写入对应餐次的 `images` 字段。一个餐次可以有多张图片，也可以没有图片（用户纯文字描述的情况）。同一餐次多张照片时，先判断是否拍的同一组食物（不同角度）还是不同食物。同一组食物取信息最丰富的那张为主，其他补充细节；不同食物各自独立识别。

#### 执行脚本

脚本依赖 Pillow。执行前先检查是否已安装，未安装时再安装：

```bash
python -c "import PIL" 2>/dev/null || pip install Pillow --break-system-packages 2>/dev/null || pip install Pillow
```

脚本位于本 skill 的 `scripts/generate_summary.py`。根据当前环境找到 skill 的安装目录，然后执行：

```bash
python <skill目录>/scripts/generate_summary.py \
  --data /tmp/food_data.json \
  --output <输出目录>/daily_summary.png
```

**关于路径的环境适配：**

- **食物图片路径**：从用户上传或当前工作目录中获取图片的实际路径，写入 JSON 对应餐次的 `images` 字段。不要硬编码特定目录。
- **输出目录**：将生成的图片保存到当前环境的合适位置，确保用户可以访问到。
- **临时文件**：JSON 数据文件可以写到系统临时目录或当前工作目录。

脚本会生成一张长图，从上到下依次是：
1. **标题栏** —— 日期和"每日饮食记录"标题
2. **汇总表格** —— 按餐次列出所有食物的名称、重量、热量，底部有合计行
3. **食物照片** —— 用户发送的原始食物图片，按餐次分组排列

生成后将图片交给用户。

## 注意事项

- 热量估算本身存在误差，这是正常的，不需要反复强调"仅供参考"。首次记录时提醒一次即可。
- 追问策略：热量影响大的差异才问用户（肉类品种、烹饪方式、饮品糖度/加料），影响小的（蔬菜品种、调味咸淡）直接取高值不打扰。
- 用户可能用很随意的方式描述食物（"那个汤"、"昨天吃的那种面"），结合对话上下文理解。
- 如果图片模糊或完全无法辨认，告知用户并请补充描述。能大致判断的情况下，按较高值估算并标注不确定。
- 一张图片中可能包含多种食物，全部识别出来。
- 食物图片路径需要从用户实际上传的位置获取，不同环境路径不同，需动态确定。
